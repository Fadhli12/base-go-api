# Implementation Plan: Tagging System

**Branch**: `015-tagging-system` | **Date**: 2026-05-12 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/015-tagging-system/spec.md`

## Summary

Organization-scoped polymorphic tagging system with CRUD, autocomplete, bulk attach/detach, and soft delete. Tags belong to organizations, attach to any taggable entity (news, invoice, media) via a join table, with slug auto-generation, usage count tracking, RBAC enforcement, and audit logging.

## Technical Context

**Language/Version**: Go 1.22+
**Primary Dependencies**: Echo v4, GORM, Casbin, PostgreSQL, Redis (permission cache)
**Storage**: PostgreSQL (UUID primary keys, partial unique indexes)
**Testing**: Go testing + testify/mock + testify/suite (unit), testcontainers-go (integration)
**Target Platform**: Linux server (Docker)
**Project Type**: REST API web service
**Performance Goals**: Standard CRUD (< 100ms p95), autocomplete < 50ms p95
**Constraints**: Follow existing codebase patterns (repository/service/handler layers, GORM soft-delete, Casbin RBAC, 9-param audit logging)
**Scale/Scope**: Org-scoped tags, ~1000 tags per org, ~10 tags per entity avg

### Key Design Decisions

1. **Organization scoping**: Follow comment system pattern — `organization_id` on both Tag and EntityTag, resolved via `resolveTagOrgDomain` helper
2. **Audit logging**: Use 9-param `LogAction` matching existing signature
3. **Pagination**: Reuse shared `ParsePagination` helper from handler
4. **Taggable types**: Hardcoded map matching `commentableTypes` — extensible later
5. **Slug generation**: Custom `GenerateSlug()` function — lowercase, regex replace, trim hyphens; numeric suffix on conflict
6. **Usage count**: Direct SQL increment/decrement with `MAX(count - 1, 0)` guard

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Slug conflict on concurrent creation | Low | Medium | Retry with numeric suffix; org-scoped partial unique index catches conflicts |
| Usage count drift | Medium | Low | Guard with `MAX(count - 1, 0)`; periodic reconciliation job (future) |
| Autocomplete performance on large tag sets | Low | Low | ILIKE with prefix index; org-scoped queries limit result set |
| Route conflict: `/tags/autocomplete` vs `/tags/:id` | Low | High | Register autocomplete route BEFORE `/:id` route |
| Taggable type extensibility | Low | Low | Hardcoded map matches comment system; future: DB-driven registry |

## Performance Considerations

- **Autocomplete**: `ILIKE` prefix search with organization_id filter — PostgreSQL B-tree index on `(organization_id, name)` handles efficiently
- **Usage count updates**: Single `UPDATE ... SET usage_count = usage_count + 1` — no read-then-write race
- **Entity tag listing**: JOIN query with GORM preloading — returns full Tag details in one query
- **Pagination**: Standard limit/offset with total count — follows existing comment/search patterns
