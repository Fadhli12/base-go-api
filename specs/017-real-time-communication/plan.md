# Implementation Plan: Real-time Communication (WebSocket)

**Branch**: `017-real-time-communication` | **Date**: 2026-05-12 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/017-real-time-communication/spec.md`

## Summary

Room-based WebSocket communication system using Hub-and-Client architecture with coder/websocket. Authenticated users join organization and entity rooms, receive live events, and broadcast presence/typing status. Redis pub/sub enables multi-instance scaling. EventBus bridge relays all existing application events to WebSocket rooms. JWT authentication on upgrade via query param. Organization-scoped presence with Redis TTL-based heartbeats. Ephemeral typing indicators. Standard WebSocket close codes for reconnection.

## Technical Context

**Language/Version**: Go 1.25+ (per go.mod)
**Primary Dependencies**: Echo v4, GORM, Casbin, Redis (go-redis/v9), coder/websocket (new)

**Testing**: Go testing + testify/mock + testify/suite (unit), testcontainers-go (integration), httptest + coder/websocket client (e2e WebSocket)
**Target Platform**: Linux server (Docker)
**Project Type**: REST API + WebSocket web service
**Performance Goals**: < 200ms message delivery p95, 1000+ concurrent connections per instance
**Constraints**: Follow existing codebase patterns (repository/service/handler layers, Casbin RBAC, structured logging via internal/logger, graceful shutdown)
**Scale/Scope**: Multi-instance via Redis pub/sub, org-scoped rooms, ~1000 concurrent connections per instance

### Key Design Decisions

1. **coder/websocket** over gorilla/websocket: Actively maintained fork with identical API, production-proven at Coder, full close code support
2. **Query param auth** over first-message auth: Simpler, standard pattern, JWT validated on upgrade. No mid-connection token rotation in v1 (reconnect required)
3. **No DB tables in v1**: All WebSocket state (presence, rooms) is in-memory + Redis. Migration 000024 is empty but reserved
4. **EventBus bridge subscribes to ALL events**: Future events auto-propagate. Room membership filters delivery — no event whitelist needed
5. **HubBroadcaster interface** to avoid circular dependency: WsEventBridge depends on `HubBroadcaster` interface (just `PublishToRoom`), not concrete `Hub`. Follows existing `SetEventBus()` setter injection pattern
6. **Room validation**: Format `org:{uuid}` or `org:{uuid}:{type}:{uuid}` where type is from existing `commentableTypes`-like constant map
7. **Presence deduplication**: Multiple connections from same user share same presence entry in Redis SET. Reference counting via INCR/DECR prevents false offline on tab close
8. **Typing as fire-and-forget**: 5s Redis TTL keys, no storage, no audit, no guarantee of delivery
9. **Hub as singleton**: One Hub instance per server process, started in `cmd/api/main.go` alongside EventBus
10. **Audit logging on REST endpoints**: `GET /presence/org/:orgID` and `GET /ws/rooms` use 9-param `LogAction` pattern for view operations
11. **Origin validation**: `WsConfig.AllowedOrigins` field (env: `WS_ALLOWED_ORIGINS`) — default `*` for dev, configurable per environment for production

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| WebSocket connection leak on abnormal disconnect | Medium | High | Read pump detects close; write pump has write deadline; Hub unregisters on both paths; heartbeat timeout catches stale connections |
| Redis pub/sub unavailable | Low | High | Local Hub continues working for single-instance; cross-instance delivery degrades; errors logged; auto-reconnect with exponential backoff |
| Room bomb (client joins 10k rooms) | Low | High | Max 50 rooms per connection enforced at join time; rooms per connection tracked in Client struct |
| Message broadcast storm (1k clients in room) | Medium | Medium | Fire-and-forget channel writes with buffer; slow clients disconnected after write deadline |
| JWT token expiry mid-connection | Medium | Low | Heartbeat checks token validity; client receives close code 4001 and must reconnect |
| Echo middleware chain compatibility | Low | High | WebSocket upgrade bypasses Echo middleware; auth extracted from query param before upgrade |
| coder/websocket API divergence from gorilla | Low | Medium | API is identical — drop-in replacement. Verify with integration tests |

## Performance Considerations

- **Message delivery**: In-process channel write per client (< 1µs). Redis pub/sub fan-out for cross-instance (< 5ms)
- **Presence heartbeat**: Redis key SET+EXPIRE every 30s per connection. Acceptable for < 10k connections
- **Typing indicators**: Redis SET+EXPIRE with 5s TTL. No accumulation, fire-and-forget to room
- **Room broadcast**: O(n) channel writes where n = clients in room. Buffered channels (256) prevent blocking
- **Memory per connection**: ~4KB (Client struct + channels + room map). 10k connections = ~40MB
